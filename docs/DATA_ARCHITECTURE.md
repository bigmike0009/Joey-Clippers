# Joe's Clippers Data Architecture

## Modeling Assumptions

- Single-tenant: one admin (`joe`) and up to ~30 members. No organization or workspace layer needed.
- All authenticated users have a row in `profiles` that is created automatically via a `after insert on auth.users` trigger.
- Role is stored on `profiles.role` (enum `admin` | `member`). There is exactly one admin; role changes are an admin-only operation.
- Slot count is per shop day, not per time slot. The app has no intra-day scheduling granularity.
- Concurrency-safe booking is enforced by a `SECURITY DEFINER` RPC and a database-level count check — not by frontend validation.
- Soft-cancellation is used for both shop days and bookings (status column) so history is preserved.
- Invite tokens are single-use UUIDs. Tokens are pre-generated by the admin and handed out out-of-band (link, text, etc.).

---

## Canonical Entities

| Entity | Description |
|---|---|
| `profiles` | One row per user; extends `auth.users` with display name and role |
| `invites` | Single-use invite tokens generated by the admin |
| `shop_days` | Days Joe declares himself open, with a fixed slot count |
| `bookings` | A member's reservation on a specific shop day |
| `day_requests` | A member's request for Joe to open on a date they suggest |

## Relational Schema

### Tables and Columns

```sql
-- profiles: extends auth.users
create table public.profiles (
  id            uuid        primary key references auth.users(id) on delete cascade,
  full_name     text        not null,
  role          text        not null default 'member' check (role in ('admin', 'member')),
  push_token    text        null,
  created_at    timestamptz not null default now()
);

-- invites: single-use registration tokens
create table public.invites (
  id            uuid        primary key default gen_random_uuid(),
  token         uuid        not null unique default gen_random_uuid(),
  email         text        null,                          -- optional pre-filled email hint
  created_by    uuid        not null references public.profiles(id),
  used_by       uuid        null references public.profiles(id),
  used_at       timestamptz null,
  expires_at    timestamptz null,                          -- null = never expires
  created_at    timestamptz not null default now()
);

-- shop_days: days Joe is open
create table public.shop_days (
  id            uuid        primary key default gen_random_uuid(),
  date          date        not null unique,
  slot_count    int         not null check (slot_count >= 1),
  status        text        not null default 'open' check (status in ('open', 'cancelled')),
  notes         text        null,
  created_by    uuid        not null references public.profiles(id),
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);

-- bookings: member slot reservations
create table public.bookings (
  id            uuid        primary key default gen_random_uuid(),
  shop_day_id   uuid        not null references public.shop_days(id) on delete cascade,
  member_id     uuid        not null references public.profiles(id),
  status        text        not null default 'confirmed' check (status in ('confirmed', 'cancelled')),
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now(),
  unique (shop_day_id, member_id)   -- one booking per member per day (any status)
);

-- day_requests: member-suggested days
create table public.day_requests (
  id              uuid        primary key default gen_random_uuid(),
  requested_by    uuid        not null references public.profiles(id),
  requested_date  date        not null,
  status          text        not null default 'pending' check (status in ('pending', 'approved', 'declined')),
  notes           text        null,
  responded_by    uuid        null references public.profiles(id),
  responded_at    timestamptz null,
  created_at      timestamptz not null default now(),
  unique (requested_by, requested_date)  -- one open request per member per date
);
```

### Keys and Constraints

| Table | PK | Unique | FKs | Check |
|---|---|---|---|---|
| `profiles` | `id` (mirrors auth.users PK) | — | `id → auth.users` | `role in ('admin','member')` |
| `invites` | `id` | `token` | `created_by → profiles`, `used_by → profiles` | — |
| `shop_days` | `id` | `date` | `created_by → profiles` | `slot_count >= 1`, `status in (...)` |
| `bookings` | `id` | `(shop_day_id, member_id)` | `shop_day_id → shop_days`, `member_id → profiles` | `status in (...)` |
| `day_requests` | `id` | `(requested_by, requested_date)` | `requested_by → profiles`, `responded_by → profiles` | `status in (...)` |

### Relationships and Cardinality

```
auth.users  1──1  profiles
profiles    1──*  invites          (admin creates many invites)
profiles    1──1  invites          (member uses one invite)
profiles    1──*  shop_days        (admin creates many shop days)
shop_days   1──*  bookings
profiles    1──*  bookings
profiles    1──*  day_requests
```

## Access Patterns and Indexing Strategy

| Query | Index |
|---|---|
| List upcoming open shop days (date ≥ today, status = open) | `idx_shop_days_date_status` on `(date, status)` |
| Count confirmed bookings for a shop day (slot availability check) | `idx_bookings_shop_day_status` on `(shop_day_id, status)` |
| List a member's confirmed bookings | `idx_bookings_member_status` on `(member_id, status)` |
| Admin: list pending day requests | `idx_day_requests_status` on `(status, created_at)` |
| Invite lookup by token (registration flow) | already covered by `unique(token)` index |

```sql
create index idx_shop_days_date_status      on public.shop_days(date, status);
create index idx_bookings_shop_day_status   on public.bookings(shop_day_id, status);
create index idx_bookings_member_status     on public.bookings(member_id, status);
create index idx_day_requests_status        on public.day_requests(status, created_at);
```

## Multi-Tenant and Authorization Model

Single-tenant. Two roles: `admin` and `member`. Role is stored in `profiles.role` and checked by all RLS policies via `get_my_role()` helper:

```sql
create or replace function public.get_my_role()
returns text language sql security definer stable as $$
  select role from public.profiles where id = auth.uid();
$$;
```

No organization layer, no workspace scoping. Every authenticated user can see all open shop days and each other's names (for the member directory). Booking details are scoped to the owner or the admin.

## Data Lifecycle

### Creation and Ownership

- **profiles**: created automatically via `after insert on auth.users` trigger from `full_name` in user metadata.
- **invites**: created only by admin via `generate_invite(email text default null)` RPC. Token UUID is generated by the database.
- **shop_days**: created only by admin via direct insert (RLS enforces admin-only writes).
- **bookings**: created only via `book_slot(shop_day_id uuid)` RPC (SECURITY DEFINER); never direct INSERT.
- **day_requests**: created by any authenticated member via direct insert (RLS allows member own-row insert).

### Updates and Concurrency

- **Slot booking race condition**: `book_slot` RPC acquires a row-level lock on the `shop_days` row using `FOR UPDATE` before counting confirmed bookings, preventing double-booking on the last slot.

```sql
create or replace function public.book_slot(p_shop_day_id uuid)
returns uuid language plpgsql security definer as $$
declare
  v_slot_count   int;
  v_booked_count int;
  v_booking_id   uuid;
begin
  -- lock the shop day row to prevent concurrent booking of last slot
  select slot_count into v_slot_count
  from public.shop_days
  where id = p_shop_day_id and status = 'open'
  for update;

  if not found then
    raise exception 'shop_day_not_found_or_closed' using errcode = 'P0001';
  end if;

  select count(*) into v_booked_count
  from public.bookings
  where shop_day_id = p_shop_day_id and status = 'confirmed';

  if v_booked_count >= v_slot_count then
    raise exception 'no_slots_available' using errcode = 'P0001';
  end if;

  -- upsert: re-confirm a previously cancelled booking on the same day
  insert into public.bookings (shop_day_id, member_id)
  values (p_shop_day_id, auth.uid())
  on conflict (shop_day_id, member_id)
  do update set status = 'confirmed', updated_at = now()
  returning id into v_booking_id;

  return v_booking_id;
end;
$$;
```

- **Cancel booking**: member can cancel their own confirmed booking; admin can cancel any booking.

```sql
create or replace function public.cancel_booking(p_booking_id uuid)
returns void language plpgsql security definer as $$
declare
  v_member_id uuid;
begin
  select member_id into v_member_id
  from public.bookings
  where id = p_booking_id and status = 'confirmed';

  if not found then
    raise exception 'booking_not_found' using errcode = 'P0001';
  end if;

  if v_member_id <> auth.uid() and get_my_role() <> 'admin' then
    raise exception 'not_authorized' using errcode = 'P0001';
  end if;

  update public.bookings
  set status = 'cancelled', updated_at = now()
  where id = p_booking_id;
end;
$$;
```

### Soft Delete / Archival / Retention

- **shop_days**: cancelled via `status = 'cancelled'`. Never hard-deleted; history is preserved.
- **bookings**: cancelled via `status = 'cancelled'`. The unique constraint `(shop_day_id, member_id)` allows the member to re-book later (upsert in `book_slot`).
- **day_requests**: status transitions to `approved` or `declined`. Never deleted.
- **profiles**: hard-deleted only when `auth.users` row is deleted (cascade). Not expected in normal operation.

## RLS Policy Plan

All tables have RLS enabled. Policies below use `auth.uid()` and `get_my_role()`.

### `profiles`
| Operation | Who | Condition |
|---|---|---|
| SELECT | authenticated | all rows (member directory) |
| INSERT | — | forbidden via RLS; trigger handles creation |
| UPDATE | authenticated | own row only (`id = auth.uid()`) |
| DELETE | — | forbidden |

### `invites`
| Operation | Who | Condition |
|---|---|---|
| SELECT | admin | all rows |
| SELECT | anon/member | `token = $token` only (used in registration flow via RPC) |
| INSERT | — | via `generate_invite()` RPC only |
| UPDATE | — | via `redeem_invite()` RPC only |
| DELETE | admin | own created invites |

### `shop_days`
| Operation | Who | Condition |
|---|---|---|
| SELECT | authenticated | all rows |
| INSERT | admin | unrestricted |
| UPDATE | admin | unrestricted |
| DELETE | — | forbidden (use status = cancelled) |

### `bookings`
| Operation | Who | Condition |
|---|---|---|
| SELECT | authenticated | own rows OR admin sees all |
| INSERT | — | via `book_slot()` RPC only |
| UPDATE | — | via `cancel_booking()` RPC only |
| DELETE | — | forbidden |

### `day_requests`
| Operation | Who | Condition |
|---|---|---|
| SELECT | authenticated | own rows OR admin sees all |
| INSERT | member | own rows (`requested_by = auth.uid()`) |
| UPDATE | admin | status + responded_by + responded_at fields |
| DELETE | — | forbidden |

## Mock Data Plan

### Seed Strategy

`supabase/seed.sql` is idempotent (uses `on conflict do nothing` or `truncate … restart identity cascade` at the top). It can be re-run safely at any time via `npx supabase db reset` or a custom `npm run db:seed` script.

### Required Dataset Shape

| Table | Row Count | Notes |
|---|---|---|
| `profiles` | 8 | 1 admin (Joe) + 7 members |
| `invites` | 10 | 7 used (one per member), 2 unused/pending, 1 expired |
| `shop_days` | 6 | 3 upcoming open, 1 upcoming with 0 slots left (full), 1 cancelled, 1 past |
| `bookings` | 12 | Mix of confirmed and cancelled across days |
| `day_requests` | 5 | 2 pending, 1 approved (has a matching shop_day), 1 declined, 1 duplicate date from different member |

### Edge-Case Records

1. **Full shop day** (`shop_days.slot_count = 3`, 3 confirmed bookings): verifies that `book_slot` returns `no_slots_available` for a 4th attempt.
2. **Cancelled shop day with existing bookings** (status = `cancelled`, associated bookings remain confirmed): verifies that members who were booked are visible to the admin even after cancellation.
3. **Member with a cancelled booking re-booking the same day**: uses the upsert path in `book_slot` — seed includes one booking row with `status = 'cancelled'` for a still-open day, allowing a test of the re-book flow.
4. **Invite token already used**: one `invites` row where `used_by` and `used_at` are set — verifies the registration flow rejects double-redemption.
5. **Day request for a date that already has an open shop day**: an `approved` day_request whose `requested_date` matches an existing `shop_days.date` — verifies the admin UI can show the relationship.
