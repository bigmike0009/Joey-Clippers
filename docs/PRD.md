# Joe's Clippers PRD

## Problem

Joe cuts hair out of his basement on no fixed schedule. Today, coordinating open days, available slots, and booking confirmations happens over a group text — chaotic for Joe and unclear for his friends. There is no single source of truth for "is Joe open, when, and can I still get in?"

## Target Users

| Role | Count | Description |
|---|---|---|
| Admin (Joe) | 1 | Barber and app owner. Controls the entire schedule: opens days, sets slot counts, approves/declines day requests, manages the member roster. |
| Member | ~20–30 | Joe's friends. View upcoming open days, book slots, cancel their own bookings, submit day requests. Added by invite only — no public sign-up. |

## Goals

1. Give Joe full, frictionless control over his schedule from his phone.
2. Give members a single place to see what's open and book a slot in one tap.
3. Eliminate back-and-forth texts for scheduling and booking confirmations.
4. Keep the member roster closed — invite-only, no public discovery.

## Non-Goals

- No payments or tipping.
- No public-facing landing page or unauthenticated browsable content.
- No recurring or auto-scheduled shop days (Joe opens each day manually).
- No multi-barber support (Joe is the only barber; no `barber_id`).
- No time-slot granularity within a day (slot count only).
- No Android support in MVP (iOS-first; Android is stretch).
- No push notification delivery in MVP (`push_token` column exists but is inert).
- No address or map features (location is always the same basement).

## Core Features (MVP Phase 1)

1. **Auth + invite-only onboarding** — Supabase email auth; members join via a single-use invite link generated by Joe. No public registration.
2. **Shop day management (admin)** — Joe creates open shop days (date + slot count), cancels them, and edits slot count.
3. **Member booking** — Members view upcoming open days and book or cancel their own slot. Backend enforces slot limits atomically.
4. **Day request flow** — Members submit a requested date; Joe sees pending requests and approves or declines each one. An approval can optionally convert directly into a new shop day.
5. **Admin booking management** — Joe can view all confirmed bookings per day and remove (cancel) any member's booking.
6. **Member directory + invite management (admin)** — Joe views the full member list, generates new invite links, and can revoke a member.

## Post-MVP Features

- Push notifications: new open day, day request response, day cancellation.
- Waitlist: auto-notify the next person when a slot opens.
- Style/cut notes per booking.
- Cut history log per member.
- Fun stats and leaderboard.
- Android support.

## Technical Design

### Stack

| Layer | Choice | Rationale |
|---|---|---|
| Mobile | React Native + Expo SDK 51+ | iOS-first, OTA updates via EAS, strong ecosystem |
| Router | Expo Router v3 (file-based) | Clean URL-style routing, typed navigation |
| Language | TypeScript (strict mode) | Catch errors at compile time, generated DB types |
| Backend | Supabase (Postgres 15 + Auth + RLS) | Managed auth, real-time capable, strong RLS story |
| State | TanStack Query v5 | Server-state caching, optimistic updates |
| UI | React Native Paper v5 | Material Design 3 baseline, well-maintained |

### Data Model

Summary — see `docs/DATA_ARCHITECTURE.md` for full schema, constraints, indexes, and RLS policies.

| Table | Purpose |
|---|---|
| `profiles` | Extends `auth.users`; holds `full_name` and `role` |
| `invites` | Single-use UUID tokens generated by admin |
| `shop_days` | Each day Joe is open (`date`, `slot_count`, `status`) |
| `bookings` | Member slot reservations; slot limits enforced by `book_slot` RPC |
| `day_requests` | Member-suggested dates; pending/approved/declined status |

Critical invariants enforced in Postgres (not frontend):
- `slot_count >= 1` on `shop_days`.
- `book_slot` RPC row-locks the shop day before counting confirmed bookings — prevents double-booking on last slot.
- One booking per member per shop day enforced by unique constraint.

### Key Screens / Surfaces

**Unauthenticated**
- Login screen (email + password)
- Register via invite (token from URL, set name + password)

**Member (tab navigator)**
- **Home** — upcoming open shop days; Book / Cancel button per day; shows remaining slots
- **My Bookings** — list of own confirmed upcoming + past bookings
- **Requests** — submit a day request; see status of own requests
- **Profile** — display name, sign out

**Admin (same tab navigator, elevated views)**
- **Home** — same as member home + Create Shop Day FAB
- **Day Detail** — list of confirmed bookings for a day; cancel-slot action per member; cancel-day action
- **Requests** — all pending requests; approve / decline actions
- **Members** — full member list; generate invite; revoke member

## Phases

### Phase 1 — MVP

Deliver all 6 core features above as individual PRs merged to `main`. App is iOS-only, invite-only, no push notifications. Acceptance criteria below must all pass.

### Phase 2

Push notifications (Expo Push), waitlist, cut notes, cut history, stats/leaderboard, Android.

---

## Acceptance Criteria

- [ ] Joe (admin) can create, view, edit, and cancel shop days from his phone.
- [ ] A member can view all upcoming open shop days and see remaining slot count.
- [ ] A member can book a slot in one tap; the slot count decrements immediately (optimistic update, confirmed by server).
- [ ] Two members simultaneously tapping "Book" on the last slot: exactly one succeeds, the other sees "No slots available."
- [ ] A member can cancel their own confirmed booking.
- [ ] Joe can cancel any member's booking from the day detail screen.
- [ ] A member can submit a day request for a specific date.
- [ ] Joe sees a pending badge on the Requests tab when new requests exist.
- [ ] Joe can approve or decline a day request; the requesting member's request status updates.
- [ ] Joe can generate an invite link and share it; a new user can register only with a valid, unused token.
- [ ] An already-used invite token cannot be used again.
- [ ] Joe can view the full member list and revoke a member.
- [ ] `npm run typecheck` passes with zero errors on main.
- [ ] Every screen handles loading, error, empty, and happy-path states.
- [ ] No hex colors, magic number spacing, or magic number font sizes outside `src/theme/`.
- [ ] No Supabase calls made directly from a component (all go through hooks → services).
